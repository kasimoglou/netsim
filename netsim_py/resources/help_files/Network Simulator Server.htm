1         Network Simulator Server

 

1.1      Installation

 

The NetSim Server has been developed and tested on recent versions of Ubuntu Server Edition (14.04 and later). A recent version (e.g., 15.04 at the time of this writing) is recommended. Although it is not a major task to port the server to other versions of Linux, such as RedHat or Debian, this is not described in the current text.

 

To install and configure the Network Simulator Server, you should perform the following steps:

 

Step 1

Install and configure the required software from ubuntu repositories

  • python3
  • python3-requests
  • python3-pytest
  • python3-psycopg2
  • python3-sqlite3
  • python3-pyproj
  • python3-scipy
  • postgresql-server
  • g++

 

Download (but do not install) OmNet++  v4.4 from https://omnetpp.org/

 

Download (but do not install)  Castalia v3.2 from https://castalia.forge.nicta.com.au/index.php/en/

 

Step 2

  • Create a user called netsim (or anythong else that you choose) for the server to run as.
  • Log in as that user.
  • Create directory   ~/dpcm_run
  • Go into ~/dpcm_run and  install omnet++  in a subdirectory.
  • Just unzip the Castalia package inside   ~/dpcm_run
  • Create a subdirectory  from the root directory (that is, /home/netsim) unpack the netsim tarball, or check out the latest stable distribution from github: https://github.com/kasimoglou/netsim

 

Step 3

  • Create a postgresql database named dpcm
  • Make user netsim the owner of this database

 

Step 4

  • Place netsim_py/src/sample.dpcm_netsim to your home directory, renaming it to .dpcm_netsim.  Edit according to the following bullets:
  • Change execdir and netsim_py to the corresponding paths of your system:
    execdir=/home/netsim/dpcm_run
    netsim_py=/home/netsim/netsim/netsim_py
  • Configure the connection to the postgesql database. If you have set up your database by the default Ubuntu installation, you should provide the following:

postgresql_connection:dbname=dpcm
In any case, if you want to use a different database, you have to provide a connection string according to the format required by python3-psycopg2. Please refer to the documentation of that package for details.

  • Choose an IP address and port the server will listen to.

gui_bind_addr=0.0.0.0

gui_bind_port=18880

  • Choose a web server according to your use:

; For development

http_server = wsgiref

; For production deployment

;http_server = cherrypy

 

Note that wsgiref is not suitable for deployment, as it does not offer any concurrency whatsoever.

 

  • Optionally, point the netsim server to a Project Repository. By default, we use the Project Repository of the WSN-DPCM project:
    project_repository = http://213.172.45.30:5984/

 

Step 5

 

Go to directory ~/netsim/netsim_py/src and execute the following commands:

source setenv

python3 setup_castalia.py

 

This step will compile the Castalia libraries and the netsim C++ runtime library.

 

Step 6

 

Run the netsim server for the first time to initialize the database. From ~/netsim/netsim_py/src type

./netsim_server  --initdb YES ll INFO

 

If all went well and you see a message called Server started, you can now use the server by visiting http://localhost:18880/admin

(or wherever you decided to serve the server).

See the instructions on using the server.

1.6      Inspecting Simulation Jobs

 

One of the main uses of the NetSim Admin Console is to inspect the status of simulation jobs, in order to help users with execution problems. Of course, as the NetSim server implementation matures,  hopefully the need for this kind of intervention will lessen. However, experience shows that it is often necessary to provide this kind of internal view into the workings of a complex system.

 

In order to understand the internal view of the system, one needs to know some details of the way the NetSim server handles simulation jobs.

 

Whenever a new simulation job is created, the system creates a home directory for this job and enters the job into the execution monitor for execution. The execution of a simulation goes through five stages:

  • PREPARE: Perform initializations for executing the simulation job.
  • GENERATE: Create the simulation by pulling into the engine all the necessary information: sensor plan, connectivity simulation outputs, node type definitions, synthesized application, etc.
  • COMPILE:  Compile any source files if needed.
  • START:  Start the event-driven simulation
  • FINISH:  Postprocess the output of the simulation and create plots and result files.

If however any of the above steps fails, the job workflow executes the ABORT step, which cleans up any stale files, but otherwise leaves unchanged the contents of the simulations home directory for inspection.

 

A simple work flow engine drives the execution, by interleaving the stages of running jobs onto a number of worker threads (depending on the processing capabilities of the underlying server). Therefore, a simulation job can be in one of three states:

·         ACTIVE:  some stage of the job is currently being executed by a worker thread.

·         READY: there are stages of the job that are waiting to be run.

·         PASSIVE: the job has finished execution (successfully or not) but the job files are still retained by the server for inspection.


 


From the Jobs console, the administrator can inspect all jobs in the system and their state. If the administrator wishes to inspect a particular job in more detail, she can click on the job name to go to the Job Inspector screen.

 

 

Here, the administrator can actually see the contents of the simulation jobs home directory, which include the inputs and outputs to the various stages of the job execution.

 

 

1.7      Inspecting the project repository

 

Besides inspecting job state, it is often useful to interact directly with the project repository, in order to solve problems related to the overall server execution (such as the code generation). In order to support this, the Admin Console offers the ability to inspect the contents of the Project Repository, and in particular interact with the the two main types of objects pertaining to simulations:

  • Network Simulation Descriptors (NSDs) and
  • VectorL modules

 

For both of these types of objects, the Admin Console offers a Browser, which displays all the objects stored in the Project Repository. Also, an administrator can execute simulation jobs using these objects, or validate them. This functionality is also available to regular users of the NetSim platform, but the administrator has the added ability to monitor the lifetime of jobs once she has started them.

 

1.7.1    Inspecting NSDs

 

The NSD inspection browser allows one to see all the NSDs visible by the NetSim server, providing also the actual Project Repository IDs (which are not available to regular users).

 

Furthermore, the administrator can validate and/or execute an NSD (create a simulation from it).

 

1.7.2    Inspecting VectorL modules

 

A similar functionality is provided for VectorL modules.

Again, the administrator is able to inspect the compilation and execution behaviour of a VectorL module.

 

 

1.8      The NSD Editor

2         The NSD Editor

 

The NSD Editor is an online tool meant to aid DPCM platform users define, access and manipulate NSD files and VectorL modules for their projects. VectorL modules are created separately from NSD files and can be attached to an NSD at a later time. Thus, we will examine separately how users can use NSD Editor in order to create new simulator descriptors and how to use it in order to create VectorL modules.

2.1        NSD Creation

 

When DPCM user is redirected to NSD Editor GUI the first screen he has access to is the one that lists his already created NSD files. In this screen the user can navigate through his saved NSDs and filter them by the project they belong to.

 

In the NSD Listing screen, the user can see for each NSD its name, the project under which this simulation descriptor was created and the plan it utilizes. Every NSD file can be cloned or deleted. By clicking on an NSD entry the user can open the corresponding NSD for editing.

 

 

In order to create a new NSD the user has to click the corresponding button placed on the top left corner of `My NSDs` screen. In the new page that he will get redirected to, he will be asked to choose a name for his NSD file and to choose the project under which this NSD will be created. In case user has been redirected to this screen by the planning tool, project will be selected automatically and it will correspond to the project currently opened in the PT.

 

 

When user submits the above information, a new NSD file is created in the database which is immediately opened for further processing.

 

User has to configure four separate sections in order to produce a full simulation: the Network section, the Parameters section, the Environment section and finally the Output section. Each part of the simulation descriptor file plays an important role to the simulation phase and we will examine each ones significance separately.

 

2.2      The Network section

 

In the Network section information related to networks set up has to be defined. Information such as, for example, number of nodes or network topology are of primary importance for the simulation process. User has already entered this information using the planning tool, so all that he has to do now is simply select the plan that is going to be used for this simulation. User will be provided only with the plan names that are created under the project that current NSD belongs to. In case user has been redirected to this screen from the PT, the currently open plan will been automatically selected. Once a plan is selected in Network section, user is able to show/hide the plans information in a hierarchical format.

 

 

2.3      The Parameters section

 

In this section user can provide the values of parameters related to the simulation process, such as simulation time limit, simulation time scale and cpu time limit.

 

 

2.4      The Environment section

 

In this section the user has to choose the model to be used for environment simulation. Environment simulation will produce realistic sensor values that are going to be used during network simulation, leading to more interesting and accurate simulation results.

 

The user can choose either a default module implementation from Castalia library or  write his own model using VectorL domain specific language and VectorL Editor. More information on VectorL syntax and language characteristics will follow on next section.

2.6      NSD Validation

 

A network simulation contains a lot of information and depends on a large number of parameters. It is not difficult for errors or omissions to enter into the provided data. Although the DPCM framework tries its best to assure that such problems do not arise, it is not feasible to foresee all possibilities.

 

Therefore, once a user has edited an NSD to his wishes, he can check its validity through the `Validation` tab. By pressing `Start NSD Validation` button, the validation process begins and any messages or errors discovered are displayed in the output panel below.

 

 

If no problems are found by the validator, a success message is displayed and the newly created NSD can be used to run successful simulations. Note however, that successful validation is not a guarantee that the actual simulation will be successful. Many things can go wrong when complex logic is executed, such as arithmetic errors, range violations in accessing data, etc. The validation step provides an additional assurance to the user, that errors detectable offline (at least those checked for) do not cause any problems later during the actual simulation.

 

 

3         The VectorL language

 

VectorL is a small but powerful domain-specific language for creating simple numerical simulations. VectorL programs are executed inside network simulations, but do not interact directly with the sensors or gateways of the system. Instead, a VectorL program is supposed to simulate some natural or artificial system (weather, pollution, car traffic, fires, etc) independent of the WSN network. However, it is possible for VectorL programs to receive input from the nodes, if the latter are equipped with actuators, that may affect the environment (e.g. lock and unlock doors, turn signal lights on/off etc).

 

A design choice for VectorL was to avoid looping constructs and data types that are not needed for  simulation. By omitting loops from the language, VectorL programs are much easier to debug and execute safely inside a simulation. On the other hand, loops are important computational components. Therefore, in order to avoid explicit loops, VectorL provides a very powerful syntax for handling arrays, similar to the syntax of mathematical software, such as Octave, yorick, or Python NumPy.

 

Even if it is small and simple, VectorL is still a programming language and our purpose here is not to provide a full documentation. A complete reference for VectorL is available together with the software distribution. In what follows, we give a brief but more or less complete presentation of the salient aspects of the language.

 

3.1      VectorL module creation and editing

 

As already was mentioned, the NSD Editor can create VectorL modules within a project. Those modules can be attached to an NSD file and will be used in the external environment simulation process. By clicking `My VectorL Files` in the NSD Editors navigation bar, the user is redirected to the list of  existing VectorL modules. This list can be filtered for a desired project, or it can show VectorL modules from all projects accessible by the user.

 

Each VectorL module has a name that is unique within the project under which it is created. Every VectorL object can be deleted. By clicking on a VectorL entry the user can access the corresponding file for editing.

 

 

 

 

In order to create a new VectorL file user has to click the corresponding button placed on the top left corner of `My VectorL Files` screen. In the form that appears, he is asked to choose a name for the new file and the project under which this file will be created.

 

 

When user submits the above information, a new VectorL file is created in the project  and it is immediately opened for further processing.

 

In the VectorL Editor screen, there is a code editor where the user can write his VectorL code. The user can at any time save his changes to the project repository.

 

 

Once finished, the user can compile his code by clicking the `Compile` button that is placed below the editor. Any information or error messages produced during compilation will be displayed in the output console.

 

 

 

3.2      Short examples

 

The simplest VectorL program has of course to be the famous hello world program. This is the code for it:

 

from sys import Init;

 

on Init {

      print("Hello world!");

}

 

Let us examine it. The first line imports name Init from module sys. Init is the name of a special event, that is emitted at the beginning of the execution of the program. The next line specifies on Init and then a print statement. This construct is an action, a statement executed when an event is triggered.

 

If we run this program, the defined action will execute at startup, print the greeting and finish. Then, as there are no more events to process, execution will end.

Let us now repeat the event count more times. We do not have loops in VectorL, so we have to either hard-code the 10 print statements, or find some other way to iterate. Indeed, we can use our own event, in order to iterate. Let us see how this can be done:

 

 

from sys import Init;

 

event greet(int count);

 

on greet

      if ( count > 0 ) {

            print("Hello world!");

           emit greet(count-1) after 1;

      }

 

 

on Init {

      emit greet(10) after 0;

}

 

There are several details to our new program. First, we declare a new event greet which takes an integer parameter, count. When a greet event is emitted, we check the count, and if it is greater than 0 we print the greeting and emit a new event.

 

It is now a good time to ask ourselves: since greet was declared here, where was Init declared? The answer of course, is that Init is declared in the sys module, which we imported into our program. This is not a special case: if another module imports our greeting module, then it can use our greet event, both emitting such events and defining actions for them.

 

The syntax of the emit statement has two parts: the event (and its parameters) to be emitted, which looks like a function call (but is very different), and an after clause. In the after clause, we specify a delay. This delay is what makes event emission different from function calls! By specifying the delay, we are introducing into our simulation the concept of  time.  Indeed, events are processed in order, based on the time of emission, and the after clause specifies that the time of emission is 1 time unit after now  (in DPCM simulations, time is always measured in seconds).

 

We can know the current time via the builtin now(). So, let us change our greeter to report the time with every greeting.

 

from sys import Init;

 

event greet(int count);

 

on greet

      if ( count > 0 ) {

            print("Hello world! The time is ",now());

           emit greet(count-1) after 1;

      }

 

 

on Init {

      emit greet(10) after 8;

}

 

If we now run this program, we get the following output:

Hello world! The time is  8.0

Hello world! The time is  9.0

Hello world! The time is  10.0

Hello world! The time is  11.0

Hello world! The time is  12.0

Hello world! The time is  13.0

Hello world! The time is  14.0

Hello world! The time is  15.0

Hello world! The time is  16.0

Hello world! The time is  17.0

 

3.3      A longer example

 

But, enough with the greeting. Now,  let us turn to something more serious: we would like to calculate something more complex. Let us simulate a parking lot! To keep things simple, we will assume that our parking lot has only two spots, and we will only simulate a few cars.

 

Our simulation has some input data, which describes, for each car, the time of arrival, time of departure and parking slot occupied.

 

// Four cars, car arrival and departure times,

//  and parking slot occupied by each car

const time carInTime = [9, 9.15, 9.30, 10] ;

const time carOutTime = [10, 9.30, 10.15, 10.30];

const int carSpot = [0,1,1,0];

 

The above code defines 3 arrays;  two of them have type time (which is just a subtype of the floating-point type real)  and the third one is an integer array with values 0 or 1 (since we only have two parking slots).

 

Note that these arrays are declared const. This tells to the compiler that the corresponding names are not variables, but named (constant) expressions. We can declare named expressions both on constant and non-constant quantities. For example, suppose we wish to declare a non-constant expression for the current time:

 

import sys;

 

def time now = sys.now();

 

We have now defined a local name, now,  which shadows the previous definition of the now() builtin. This is perfectly legal, and we do not run the danger of losing the initial now() builtin, since it is available through the system module, sys.

 

We also need to keep the state of each parking spot. For this, we define an actual variable:

 

/* The state of the parking spots */

var bool spotTaken = [false, false];

 

This array is of type Boolean. The observant reader will note that we did not declare the size of our array, nor did we somehow indicate that spotTaken is in fact an array type instead of a scalar.  This is because, in VectorL, there is no distinction: everything is an array, although of course scalars are zero-dimensional arrays.

 

We also introduce another variable (although it is redundant),  to help us with the simulation: a variable that iterates over the car arrivals.

 

var int arrival = 0;

Now, we are ready to write the logic of our simulation. We shall define two events

 

event carIn(int i);

event carOut(int i);

 

and the corresponding actions

 

on carIn {

    def int pos = carSpot[i];

 

    print("Car arrived at spot", pos, "at time=", now);

    spotTaken[pos] := !spotTaken[pos];

 

    // schedule departure

    emit carOut(arrival) after carOutTime[arrival]-now;

 

    // schedule arrival of next car

    arrival := arrival + 1;

    if(arrival < shapeof(carInTime)[0]) {

       emit carIn(arrival) after carInTime[arrival]-now;

       print("Next arrival will be at time",

            carInTime[arrival]);

    } else {

       print("No next arrival, total arrivals=",arrival);

    }

}

 

on carOut {

    def int pos = carSpot[i];

 

    print("Car left spot", pos, "at time=", now);

    spotTaken[pos] := !spotTaken[pos];

}

 

 

Putting everything together and running the example, we have the following output

 

Car arrived at spot 0 at time= 9.0

Next arrival will be at time  9.15

Car arrived at spot 1 at time= 9.15

Next arrival will be at time  9.3

Car left spot 1 at time= 9.3

Car arrived at spot 1 at time= 9.3

Next arrival will be at time  10.0

Car left spot 0 at time= 10.0

Car arrived at spot 0 at time= 10.0

No mext arrival, total arrivals= 4

Car left spot 1 at time= 10.15

Car left spot 0 at time= 10.3

Now,  suppose that would like to run both our parking simulation and our greeting simulation. We could of course copy the two codes into one file manually. However, this is not necessary in VectorL; it was designed specifically to make model combination very easy. All we have to do is write a very simple module that includes the previous two ones:

 

// Include both greetings2 and parking

3.6      Semantics of modules

 

A VectorL program is constructed by a number of included modules. Modules are included in the program by import statements mentioned in other modules, with the exception of a base module, that is specified by the user to the compiler.

 

A module A imports module B, if B appears in an import or from statement in the text of A. Cyclical importation is a compile-time error. Therefore, the import relation between modules can be extended (by taking its transitive closure) to a partial order.

 

The set of included modules in a program is defined as the set of modules (transitively) imported by the base module (note that this definition includes the base module itself). Furthermore, the special sys module is always included in the program. The inclusion order is a total order, determined by a depth-first traversal of the import graph. The order of inclusion is determined according to the following rules:

·         If module A imports module B, then B is included in the program before A.

·         If some module imports module A and then module B in its text, and including A does not cause B to be included, then A will be included before B.

Note that the sys module is always first in the inclusion order.

3.7      Semantics of expressions

 

Where VectorL differs from most conventional languages is in that all its values are array values.  In VectorL, a value has a shape, which is determined at compile time. The shape is a (possibly empty) sequence of positive integers. Those values whose shape is the empty list are called scalars. The length of the shape of an expression is called the order of the expression (borrowing the terminology from tensors).

 

All operators operate on their operands in a pointwise manner. When operands have equal shape, the semantics are clear. When the shapes differ, the operands are checked to see if they are broadcastable. If so, then the operator is applied pointwise on the broadcast values. If not, a compile-time error is signaled.

 

Two arrays are shape-compatible if they are of the same order  and  the corresponding array dimensions are either equal to each other, or one of them is equal to 1.  When two arrays have different order, the one of less order can be brought to have order equal to the others by prepending 1s to its shape. Two arrays are broadcastable iff, after bringing them to the same order, they are shape-compatible. Broadcasting is commutative and associative, so many arrays can be broadcast together.

 

For example, a shape (3,4) is broadcastable with the following shapes:  (4), (2,3,4), (3,1), and ().  It is not broadcastable with shapes (3), (2,4), or (4,3).  Note that scalars are broadcastable to any shape.

 

Except from the standard C/C++ operators, VectorL also supports three other, array-specific operators.

3.7.1    Array construction

 

The expression [ E1,  E2,  , En ], where Ei are all broadcastable to shape s,  is a new array with shape equal to prepending n to s. This operator is the standard array constructor. For example [1,2,4]  is an array of shape (3).  [1, [1,2], [3,4]] is an array of shape (3,2). Note that in the last example, the first operand was broadcast.

 

3.7.2    Concatenation

 

Concatenation is done only on non-scalar arrays.  The concatenation of n arrays is written as  ( E1,  E2,  , En ).  The shapes of the operands have to be compatible in every dimension except the first. Then, the concatenation operator concatenates the operands on the first dimension. For example,  ( [1,2] , [3,4] ) is the array [1,2,3,4].  Another example:  if  A is an n-vector of coordinates, then (A, [1]) is the n+1-vector of homogeneous coordinates.

 

3.7.3     Indexing

 

The indexing operator is a powerful way to both alter the shape of an array and obtain a subset of its values.  We shall start with a few examples, where we assume that A is a (5,4)-shaped array.

 

A[2]                 is a (4)-shaped array containing row 2 of A.

A[4,3]              contains the bottom-right element of array A.

A[0:3, 3]         is a (3)-shaped array containing the first 3 elements of column 3

A[:,:]                is the array A itself.

A[,_]            is a  (5,4,1)-shaped array, with contents equal to A.

 

Those familiar with the numerical library NumPy, should recognize the above examples. We suggest that you read the excellent documentation on NumPy in order to exploit the power of this notation.

 

Formally, an indexer on an order-k array of shape (n1, n2, ,  nk)  is a sequence of indexing operators. Each indexing operator, with the exception of the ellipsis ,
corresponds to one dimension of the indexed array. Each indexing operator can one of the following:

 

i           where i is scalar, selects the corresponding element on this dimension, and deletes the dimension from the results shape.

p:q:s    where p, q, s are scalars, defines a slice of the array on this dimension. S is the stride, and it may be negative, but not zero. When either is omitted, the defaults are:

·         For positive stride, p=0 and q=<dimension>.

·         For negative stride, p=<dimension>-1 and q=-1.

·         An omitted stride defaults to 1.

_          designates a  new dimension of length 1 to be inserted in the shape list of the result

        the ellipsis matches a number of dimensions, so that one can write indexing expressions for arrays of unknown order. This is useful in named expressions, among others.

 

 

In some rare cases, the compiler may signal an error to a syntactically legal indexing operator. This happens if the compiler cannot infer the size of the indexing expression for some dimension.

 

For example,  A[i:j] where i and j are variables, is syntactically legal, but the compiler cannot infer the shape of the result at compile time. However,  A[i:i+4] has a well-defined shape that the compiler will infer.

 

3.8      Named expressions

 

Named expressions are more like macros and not really function-like. They are introduced with the keywords  def or const (when the expression has no parameters).  Notice that a const is semantically identical to a def; the compiler will infer (and exploit) compile-time const-ness of any named expression (or any expression anywhere for that matter).

 

A named expression without parameters can be tricky (like macros sometimes are). For example, assume the following code

 

var int i = 0;

def isquare = i*i ;

 

{

   

   i  := 2;

   // now isquare is 4

   i  := i+1;

   //  now isquare is 9

}

 

A named expression can also be defined in a block manner:

 

def real binomial_roots(real a, real b, real c)

{

    def real D = b*b  4*a*c;

    def real sqrtD = sqrt(D);

          

    // return expression

    (-b + [sqrtD, -sqrtD]) /(2*a)

}

 

This named expression uses locally defined named expressions in its computation. It can be called on any triple a,b,c of broadcastable arrays and will compute all binomial roots in parallel.

 

Note that the above expression may fail, if some discriminant is negative! Therefore, a more robust implementation should be used in practice:

 

def real binomial_roots(real a, real b, real c)

{

    def real D = b*b  4*a*c;

    def real sqrtD = sqrt( (D>=0) ? D : nan );

          

    // main expression computed

    (-b + [sqrtD, -sqrtD])/(2*a)

}

 

3.9      Semantics of statements

 

The semantics of statements in VectorL are quite simple, so no elaborate description is needed.

 

3.9.1    Emit statements

 

An emit statement emits a new event, which will be triggered (i.e., its actions will be executed) with a desired delay, in terms of simulation time. Events are objects that can only take scalar parameters. Let us emphasize this point:  event parameters must be scalar. This restriction is not so serious in practice, as events do not have to carry data to their actions, only information about which data to process.

 

The time specified in the after clause of the emit statement is the trigger delay, that is, the delay after which the event will be dispatched, and must be a scalar value. If the event should be dispatched immediately, zero (or even a negative value) can be provided.

3.9.2    If statements

 

The if statement takes a scalar value in the parentheses and, similar to C or C++ evaluates its truthiness by comparing it to zero.

3.9.3    Assignments

 

Assignments in VectorL are similar in concept to assignments in other languages. However, the array semantics of VectorL make assignments more powerful.

 

As in other imperative programming languages, assignment copies data from an expression on the right-hand side of the assignment, to an l-value expression on the left-hand side of the assignment.

 

It is a compile-time error if the right-hand sides shape is not broadcastable to the shape of the left-hand side.

 

In VectorL, an  l-value is defined as follows:

·         A named variable (defined in any module) is an l-value.

 

·         If  A  is an l-value and I=[I1, I2, , In] any indexing operator, then the expression  A[I1, I2,  In] is an l-value.

 

Thus, one can write, for example,   A[::-1] := A to reverse the contents of an array. Of course, the same effect can be achieved by the (more conventional)  assignment  A :=  A[::-1]

 

·         If A1, A2, , An are l-values with equal (not just broadcastable) shapes, then the concatenation expression  (A1, A2, , An)  is an l-value.

 

Therefore, one can write  x,y := y,x to swap the  contents of variables x and y.

 

3.9.4    Print statement

 

The print statement is used to output information to the standard output file. It can accept an arbitrary number of operands. Operands can be arrays of any type and shape, or they can be string literals. Note that VectorL does not support[1] character strings as a normal type. String literals may only appear as arguments in print statements.

3.9.5    Block statement

 

A block statement is a possibly empty list of statements and named expression declarations. As in other programming languages, named expressions defined in a block statement are only valid in the scope of the block statement.

3.10 Semantics of actions

 

An action is a statement that is executed when the corresponding event is emitted. Any number of actions can be declared for an event, and in any module. The statement of an action is usually a block statement, but this is not necessary. For example, the database idiom of Event-Condition-Action can be written as

 

on Event if ( Condition ) {  /* Action */  }

 

When an event is triggered, all of  its actions are executed. The order of  execution is fixed and corresponds to the import relationship between the modules in which the actions are defined. The rules are as follows:

 

·         If two actions are defined in the same module, then they are triggered in the order they are defined.

·         If module A is included in the program before module B, then any action defined in A is executed before any action defined in B.

 

However, in a well-written program, it is not a good idea to depend too much on the execution order of actions. Indeed, it is desirable that actions be logically independent.

 

The statement of an action is interpreted inside the scope of the module it is defined in. In particular, an action statement does not have implicit access to any declared expressions or variables of the model where the event is defined. In addition, the actions statement has within its scope the parameters of the processed event. These parameters override any declarations of the same name in the scope where the action is defined.
